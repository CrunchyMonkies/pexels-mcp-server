# Pexels MCP Server — Gap Analysis (2026-06-02)

Comparison of the server against the official
[Pexels API documentation](https://www.pexels.com/api/documentation/).

> **Update:** Most gaps below were **resolved** in the v2.0.0 Bun migration. The only
> remaining gap is #1 (My Collections), which is deferred because it requires OAuth 2.0.
> See [`specification.md`](./specification.md) for the current behaviour.

## Summary

The server covers all **public, API-key-accessible** Pexels endpoints. As of v2.0.0 it also
adds runtime response validation (zod), a 429 retry/backoff, image-manipulation tools, and a
Streamable HTTP transport. The single outstanding gap is the OAuth-gated "My Collections".

## Gaps & resolution status

| # | Area | Status | Resolution |
|---|---|---|---|
| 1 | **My Collections** (`GET /collections`, OAuth 2.0) | ⏸️ **Deferred** | Still not exposed. `PexelsService.getMyCollections` is retained (with a doc note) but no tool is registered. Requires implementing Pexels OAuth 2.0 — tracked as future work. |
| 2 | **`per_page` default** undocumented | ✅ Resolved | All list-tool descriptions now state "1-80 (Pexels default is 15)". |
| 3 | **Pagination URLs not surfaced** | ✅ Resolved | List tools append a "Pagination: … use the `page` parameter" hint when `next_page`/`prev_page` are present, and the URLs remain in the JSON. |
| 4 | **Video vs photo `size` semantics** | ✅ Resolved | `searchVideos.size` description now reads "large (4K), medium (Full HD), small (HD)"; photo size notes "24MP/12MP/4MP". |
| 5 | **HLS video quality** | ✅ Resolved | `downloadVideo.quality` enum is now `hd` \| `sd` \| `hls`, with documented fallback. |
| 6 | **No retry/backoff on 429** | ✅ Resolved | `PexelsService.request` retries up to 2× on HTTP 429, honouring `Retry-After` / `X-Ratelimit-Reset` (capped at 10s). Covered by unit tests. |
| 7 | **Resource envelope wrapping** | ✅ Resolved | The three resource handlers now return unwrapped `data`, matching tool output. |
| 8 | **API drift vs live docs** | ✅ Mitigated | All response object schemas use zod `.passthrough()`, so new Pexels fields are preserved rather than rejected; responses are now validated at runtime. |

## New capabilities (v2.0.0 → v2.1.0)

- **Image pipeline** (v2.1.0): an ordered `pipeline` array on `getPhoto`/`downloadPhoto` with ops
  `resize`/`cut`/`rotate`/`flip`/`flop`/`modulate`/`grayscale`/`format`/`placeholder`, via
  `Bun.Image` (`cut`/crop uses `fast-png`). Replaces the standalone `transformPhoto`/
  `generatePhotoPlaceholder` tools.
- **LRU + TTL cache** (v2.1.0) for source images and processed results
  (`IMAGE_CACHE_MAX`/`IMAGE_CACHE_TTL_MS`).
- **Transport-gated capabilities** (v2.1.0): `downloadPhoto`/`downloadVideo` and file output
  (`outputPath`) are available only over the **stdio** transport.
- **Streamable HTTP transport** (v2.0.0) alongside stdio (`MCP_TRANSPORT=http`), with sessions.
- **Zod response validation** (v2.0.0) for every Pexels endpoint.
- **Unit test suite** (`bun test`) covering the service, schemas, cache, image pipeline, and server.
- **Agent Skill** (v2.1.0) under `skills/pexels-mcp/` describing how to drive the server.

"My Collections".
